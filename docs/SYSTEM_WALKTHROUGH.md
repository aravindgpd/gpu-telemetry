# GPU Telemetry Pipeline — Complete System Walkthrough

## 1. The Big Picture

You're building a **horizontally-scalable telemetry pipeline** for a GPU-equipped AI cluster. The spec demands a custom message queue (no Kafka/Rabbit/ZMQ), 4 services, Helm-deployable on Kubernetes, with every layer scalable up to 10 replicas.

```
                ┌─────────────────────────────────────────────┐
                │           AI Cluster (concept)              │
                │   N hosts × M GPUs each, DCGM exporter      │
                │   produces CSV-formatted metric rows        │
                └────────────────────────┬────────────────────┘
                                         │ (CSV file in our case)
                                         ▼
   ┌─────────────┐    proto bytes    ┌──────────────┐    proto bytes    ┌────────────┐    SQL    ┌────────────┐    HTTP    ┌────────┐
   │  Streamer   │ ════════════════► │ Message      │ ════════════════► │ Collector  │ ════════► │ PostgreSQL │ ◄═════════ │Gateway │ ──► User
   │ (StatefulSet│   gRPC bidi       │ Queue Broker │  gRPC server      │ (Deployment│           │            │            │ (REST) │
   │  ≤ 10 pods) │   stream          │ (StatefulSet)│  stream           │  ≤ 10 pods)│           │            │            │        │
   └─────────────┘                   └──────────────┘                   └────────────┘           └────────────┘            └────────┘
```

**Three big design decisions** (matching the interviewer's clarifications):
1. The CSV's timestamp column is decorative — the **Streamer's wall-clock at publish time** is the canonical timestamp.
2. **One shared Postgres** for all Collector replicas, with idempotency from a `UNIQUE(uuid, metric_name, sample_at)` constraint.
3. **Row-based partitioning** of the single CSV across Streamer replicas — `row_num % STREAMER_TOTAL == STREAMER_INDEX`.

[broker.go:88](../services/messagequeue/internal/broker/broker.go#L88) — `broker.Publish` looks up the topic, picks the partition, then calls into the partition. Inside [partition.go:67-105](../services/messagequeue/internal/broker/partition.go#L67):
```go
func (p *partition) Append(payload, headers, timestamp) (int64, error) {
    p.mu.Lock(); defer p.mu.Unlock()
    offset := p.head                            // assign next offset (e.g. 47)
    p.wal.Append(walRecord{Offset: 47, ...})    // 1) durable WAL write FIRST
    idx := offset % int64(p.capacity)           // 2) compute slot index (47 % 65536)
    p.slots[idx] = slot{offset: 47, payload, ...}  // 3) ring buffer write
    p.head++
    if p.head - p.tail > capacity {             // 4) advance tail if we wrapped
        p.tail = p.head - capacity
    }
    for s := range p.subs {                     // 5) wake subscribers (non-blocking)
        select { case s.notify <- struct{}{}: default: }
    }
    return 47, nil
}
```

#### Stage 8b: Cross-GPU search via `/api/v1/telemetry`

For "GPU_UTIL across every H100 in the last 5 minutes", the per-GPU endpoint would require N round trips. Instead:

```bash
curl 'http://localhost:8080/api/v1/telemetry?metric_name=DCGM_FI_DEV_GPU_UTIL&model_name=h100&limit=200'
```

[gpu.go QueryTelemetry handler](../services/gateway/internal/handler/gpu.go) calls `db.QueryTelemetry(filter)`, which runs:

```sql
SELECT ts.id, ts.uuid, ts.metric_name, ts.ingested_at, ts.sample_at, ts.value, ...
FROM telemetry_samples ts
JOIN gpus g ON ts.uuid = g.uuid
WHERE ($1::text        IS NULL OR ts.uuid        = $1)
  AND ($2::text        IS NULL OR ts.metric_name = $2)
  AND ($3::text        IS NULL OR g.model_name  ILIKE $3)   -- ← substring match
  AND ($4::timestamptz IS NULL OR ts.sample_at  >= $4)
  AND ($5::timestamptz IS NULL OR ts.sample_at  <= $5)
ORDER BY ts.sample_at ASC
LIMIT $6 OFFSET $7
```

Two design notes:

1. **ILIKE substring for `model_name` and `hostname`.** The repo wraps user input in `%...%` via the `likeArg` helper ([gateway/store/postgres.go](../services/gateway/internal/store/postgres.go)). So `?model_name=h100` matches `"NVIDIA H100 80GB HBM3"`. The user input is passed as a bound parameter — no SQL injection. UUIDs and metric names stay exact-match because they're stable identifiers.

2. **JOIN against `gpus`.** Required because `telemetry_samples` doesn't carry `model_name`. The unconditional JOIN is cheap at our scale (≤ a few thousand GPU rows); at production scale you'd denormalise `model_name` into the fact table.

### 4.1 The WAL — durability layer ([wal.go](../services/messagequeue/internal/broker/wal.go))

**What**: an append-only file per partition, binary-encoded records, replayed on startup.

**Format**:
```
+──────────+────────+──────────────+──────────+──────────+──────────────────+
│ body_len │ offset │ payload_len  │ payload  │ n_hdrs   │ key/val pairs    │
│  uint64  │ int64  │   uint32     │  bytes   │  uint32  │ (4B+key+4B+val)* │
+──────────+────────+──────────────+──────────+──────────+──────────────────+
```

**Key methods**:
- [wal.go:60-103](../services/messagequeue/internal/broker/wal.go#L60) `Append(rec)` — writes one record, flushes to OS buffer. fsyncs every `MQ_WAL_SYNC_BYTES` (default 4096) for crash safety vs. throughput trade-off.
- [wal.go:127-167](../services/messagequeue/internal/broker/wal.go#L127) `replayWAL(...)` — reads every record at startup, hands them to a callback. Returns the next offset to assign (one past the highest seen).
- The `body_len` prefix lets the reader recover from a partial trailing record (crashed mid-write) — once `io.ReadFull` returns short, we stop cleanly.

### 4.2 The partition — in-memory layer ([partition.go](../services/messagequeue/internal/broker/partition.go))

**Data structure** (lines 31-40):
```go
type partition struct {
    id       int32
    capacity int            // e.g. 65536, must be power of 2
    mu       sync.Mutex     // guards everything below
    slots    []slot         // ring buffer
    head     int64          // next offset to assign
    tail     int64          // oldest offset still in memory
    subs     map[*subscriber]struct{}  // who to notify on publish
    wal      *walWriter
}
```

**The `Append` flow** ([partition.go:67-105](../services/messagequeue/internal/broker/partition.go#L67)):
1. Lock the partition (single-writer per partition)
2. WAL write FIRST (durability before memory)
3. Compute `idx = offset % capacity` — the ring slot
4. Overwrite `slots[idx]` with the new message (anything that lived there is gone from memory; still in WAL)
5. Advance `head++`; if `head - tail > capacity` advance `tail` (we wrapped, oldest data is no longer in memory)
6. Wake all subscribers via non-blocking sends on their `notify` channels — coalesced wake-up, never blocks the writer

**The `Read` flow** ([partition.go:108-128](../services/messagequeue/internal/broker/partition.go#L108)):
- `offset < tail` → `errOffsetTooOld` (consumer fell behind ring buffer)
- `offset >= head` → `errOffsetFuture` (consumer caught up; no new data)
- otherwise → `slots[offset % capacity]`, with a safety check that `s.offset == requestedOffset` (defense against torn reads)

**On startup** ([partition.go:46-65](../services/messagequeue/internal/broker/partition.go#L46)) `newPartition` takes the replayed WAL records and reseeds the ring buffer + sets head/tail correctly. So after a restart, in-memory state matches the on-disk truth.

### 4.3 The consumer group — bookmark layer ([group.go](../services/messagequeue/internal/broker/group.go))

**State** (lines 18-36):
```go
type consumerGroup struct {
    members     map[string]*groupMember   // by consumer_id
    assignments map[int32]string          // partition → owner consumer_id
    committed   map[int32]int64           // partition → highest acked offset
}

type groupMember struct {
    id   string
    done chan struct{}  // closed → "you're rebalanced; reconnect"
}
```

**Round-robin assignment** ([group.go:151-166](../services/messagequeue/internal/broker/group.go#L151)):
```go
func computeAssignments(members map[string]*groupMember, numPartitions int32) map[int32]string {
    ids := sorted_keys(members)               // deterministic order
    for p := int32(0); p < numPartitions; p++ {
        out[p] = ids[int(p) % len(ids)]
    }
}
```

**Stable rebalance** — only members whose set actually changed get evicted ([group.go:64-94](../services/messagequeue/internal/broker/group.go#L64)):
```go
oldByMember := invert(g.assignments)
g.assignments = computeAssignments(g.members, g.numPartitions)
newByMember := invert(g.assignments)
for id, m := range g.members {
    if id == newJoiner { continue }
    if !equalInt32Sets(oldByMember[id], newByMember[id]) {
        closeOnce(m.done)                    // tell this member to reconnect
        evicted = append(evicted, m)
    }
}
```

This is the key "no thrashing" property: when the same fleet reconnects after a churn event, the `computeAssignments` function returns the same map, so subsequent Joins don't trigger more evictions.

**Grace-period leave to avoid double-rebalance.** Without further protection, the rebalance signal would cause a second problem: when a Subscribe handler exits because its `done` channel closed, calling `group.Leave(memberID)` immediately would recompute assignments AGAIN as if the member had disappeared — kicking off a cascading rebalance. The fix has two parts:

1. **[Subscription.SkipLeaveOnCleanup()](../services/messagequeue/internal/broker/broker.go#L302)** — the gRPC handler sets this flag right before returning when the exit reason is a rebalance signal (not a real disconnect). `Cleanup()` then unregisters from partitions but does NOT call `group.Leave` immediately.
2. **[scheduleStaleEviction()](../services/messagequeue/internal/broker/broker.go#L330)** — a `time.AfterFunc(30s, ...)` callback. If the consumer reconnects within 30 seconds, `Join` replaces the `*groupMember` pointer, so the timer's pointer-equality check ([group.go:110-118](../services/messagequeue/internal/broker/group.go#L110)) sees a mismatch and bails out — no eviction. If the consumer never returns (pod crash + replaced by a pod with a different consumer_id), the timer fires and properly removes the stale entry.

```go
// group.go — the pointer-equality guard
func (g *consumerGroup) LeaveIfSameMember(memberID string, expected *groupMember) (bool, []*groupMember) {
    cur, ok := g.members[memberID]
    if !ok || cur != expected {
        return false, nil    // ← consumer re-Subscribed; new pointer; abort
    }
    return true, g.leaveLocked(memberID)
}
```

Without this two-layer protection, every rebalance triggered a follow-up rebalance, doubling churn. Verified by `TestGroupLeaveIfSameMemberWhenReplaced` and `TestGroupLeaveIfSameMemberWhenStill` in [group_test.go](../services/messagequeue/internal/broker/group_test.go).

### 4.4 The broker orchestrator ([broker.go](../services/messagequeue/internal/broker/broker.go))

The top-level facade that the gRPC service talks to. Three principal methods:

- [broker.go:48-83](../services/messagequeue/internal/broker/broker.go#L48) `CreateTopic(name, partitions)` — replays WAL for each partition, opens new WAL writers, builds partitions, registers in `b.topics`. Idempotent.
- [broker.go:88-104](../services/messagequeue/internal/broker/broker.go#L88) `Publish(topic, partition, payload, headers)` — looks up the topic, applies round-robin if `partition < 0`, calls `Append` on the partition.
- [broker.go:108-156](../services/messagequeue/internal/broker/broker.go#L108) `Subscribe(topic, group, memberID)` — gets/creates the group, calls `group.Join`, registers a single `subscriber` on every owned partition (so any publish to those partitions wakes the same notify channel), initialises read cursors from committed offsets, returns a `Subscription` handle.

**The `Subscription` handle** ([broker.go:181-258](../services/messagequeue/internal/broker/broker.go#L181)) is what the gRPC stream handler uses to actually pump data:
- `Notify()` — returns the wakeup channel
- `Done()` — returns the rebalance channel
- `PollOnce()` — reads one message from each owned partition at the current cursor, returns them in a slice
- `Cleanup()` — unregisters from partitions and removes the member from the group

### 4.5 The gRPC service ([service.go](../services/messagequeue/internal/server/service.go))

Implements `mqpb.MessageQueueServer`. Each method is a thin shell over a broker call:

| RPC | Type | Body |
|---|---|---|
| `CreateTopic` | unary | `broker.CreateTopic` |
| `Publish` | bidi stream | recv loop → `broker.Publish` → send response |
| `Subscribe` | server stream | `broker.Subscribe` → main loop: `PollOnce` → `stream.Send`, block on `Notify`/`Done`/`ctx` |
| `Acknowledge` | unary | `broker.Acknowledge` |
| `GetOffsets` | unary | `broker.GetOffsets` |
| `HealthCheck` | unary | static `{ready: true}` |

The Subscribe loop ([service.go:78-127](../services/messagequeue/internal/server/service.go#L78)) is the most interesting:
```go
for {
    delivered := sub.PollOnce()
    for _, d := range delivered { stream.Send(&DeliveryMessage{...}) }
    if len(delivered) > 0 { continue }    // hot-loop while there's data
    select {                              // otherwise wait
    case <-ctx.Done():   return nil
    case <-sub.Done():   stream.Send(&DeliveryMessage{RebalanceSignal: true}); return nil
    case <-sub.Notify(): continue
    }
}
```

This balances throughput (continue draining when there's data) with efficiency (block on signals when there isn't). The non-blocking notify on the publisher side means publishing is never delayed by slow consumers.

## 5. Storage Layer

### 5.1 Schema (long/melted format)

[migrations/0001_initial_schema.up.sql](../services/collector/internal/store/migrations/0001_initial_schema.up.sql):

**`gpus`** — dimension table, one row per physical GPU, keyed by UUID:
```sql
uuid       VARCHAR(64)  PRIMARY KEY,
gpu_index  VARCHAR(8),       -- "0".."7" per host
device     VARCHAR(32),      -- "nvidia0"
model_name VARCHAR(128) NOT NULL,
hostname   VARCHAR(128),
created_at TIMESTAMPTZ  DEFAULT NOW(),
updated_at TIMESTAMPTZ  DEFAULT NOW()
```

**`telemetry_samples`** — fact table, one row per `<uuid, metric, sample_at>`:
```sql
id           BIGSERIAL PK,
uuid         VARCHAR(64)  NOT NULL REFERENCES gpus(uuid),
metric_name  VARCHAR(64)  NOT NULL,    -- e.g. DCGM_FI_DEV_GPU_UTIL
ingested_at  TIMESTAMPTZ  NOT NULL,    -- streamer publish time
sample_at    TIMESTAMPTZ  NOT NULL,    -- canonical timestamp (= ingested in current design)
value        DOUBLE PRECISION NOT NULL,
container, pod, namespace, labels_raw,
UNIQUE (uuid, metric_name, sample_at)  -- THE idempotency key
```

Three indexes for query patterns:
- `(uuid, sample_at DESC)` — "latest samples for GPU X"
- `(metric_name, sample_at DESC)` — "all GPUs by metric"
- `(uuid, metric_name, sample_at DESC)` — "specific GPU's specific metric"

### 5.2 Migrations ([migrate.go](../services/collector/internal/store/migrate.go))

Embedded SQL files via `//go:embed migrations/*.up.sql`. Each migration runs in a transaction with a sentinel insert to `schema_migrations`:

```go
tx.Exec(ctx, sqlBody)                                    // the migration SQL
tx.Exec(ctx, `INSERT INTO schema_migrations VALUES ($1)`, version)
tx.Commit()
```

Failure rolls back both. Lexicographic sort over `0001_*.up.sql`, `0002_*.up.sql`, ... gives deterministic apply order.

### 5.3 Idempotent writes

[postgres.go:82-98](../services/collector/internal/store/postgres.go#L82) — `UpsertGPU`:
```sql
INSERT INTO gpus (...) VALUES (...)
ON CONFLICT (uuid) DO UPDATE SET
   gpu_index = EXCLUDED.gpu_index, ...
   updated_at = NOW()
```
GPU dimension always reflects most recently observed state.

[postgres.go:102-117](../services/collector/internal/store/postgres.go#L102) — `InsertTelemetry`:
```sql
INSERT INTO telemetry_samples (...) VALUES (...)
ON CONFLICT (uuid, metric_name, sample_at) DO NOTHING
```
Re-delivered messages are silently no-op'd. **At-least-once MQ delivery + idempotent insert = effectively exactly-once.**

## 6. The Proto Contracts

### 6.1 [telemetry.proto](../proto/telemetry/telemetry.proto)

The payload that flows through the MQ:
```proto
message TelemetryRecord {
  int64  ingested_unix_ns = 1;   // Streamer publish time (canonical)
  int64  sample_unix_ns   = 2;   // = ingested_unix_ns currently
  string metric_name      = 3;
  string uuid             = 4;
  string gpu_index        = 5;
  string device           = 6;
  string model_name       = 7;
  string hostname         = 8;
  string container        = 9;
  string pod              = 10;
  string namespace        = 11;
  double value            = 12;
  string labels_raw       = 13;
}
```

After `proto.Marshal`, this is roughly 80–200 bytes per sample. The MQ doesn't care what's inside — it just shuttles `bytes payload`.

### 6.2 [mq.proto](../proto/mq/mq.proto)

Six RPCs. Of note:

**`Publish` (bidi stream)** — producer keeps the connection open and pipelines requests:
```proto
rpc Publish(stream PublishRequest) returns (stream PublishResponse);
```
Request has `topic, partition, payload, headers`; response has `partition, offset, error`.

**`Subscribe` (server stream)** — consumer issues one request, broker pushes forever:
```proto
rpc Subscribe(SubscribeRequest) returns (stream DeliveryMessage);
```
`DeliveryMessage` has both **data fields** (partition, offset, payload, headers, timestamp_ns) and **control fields** (rebalance_signal, assigned_partitions). One channel, two purposes.

**Acknowledge / CreateTopic / GetOffsets / HealthCheck** — standard unary calls.

## 8. Cross-cutting Concerns

### 8.1 Logging — `go.uber.org/zap`

Every service uses zap for structured JSON logs. Examples:

- Streamer reader logs each published row at debug level + a warning per skipped malformed row ([reader.go:135-139](../services/streamer/internal/reader/reader.go#L135))
- Collector consumer logs subscription, errors, rebalance signals ([consumer.go:74-77](../services/collector/internal/consumer/consumer.go#L74), [consumer.go:97-101](../services/collector/internal/consumer/consumer.go#L97))
- MQ broker logs topic creation, member subscribe, eviction, ring overflow ([broker.go:74-77](../services/messagequeue/internal/broker/broker.go#L74), [broker.go:139-142](../services/messagequeue/internal/broker/broker.go#L139))
- Gateway has zap middleware logging every HTTP request with method, path, status, duration ([middleware.go:11-31](../services/gateway/internal/handler/middleware.go#L11))
- MQ has gRPC interceptors logging every RPC ([interceptors.go:11-35](../services/messagequeue/internal/server/interceptors.go#L11))

### 8.2 Graceful shutdown

Every `cmd/server/main.go` follows the same pattern:
```go
ctx, stop := signal.NotifyContext(context.Background(), syscall.SIGTERM, syscall.SIGINT)
defer stop()

// ... start server in a goroutine ...

<-ctx.Done()                     // block until SIGTERM/SIGINT
// graceful shutdown (varies per service)
```

- Gateway: `srv.Shutdown(timeoutCtx)` — drains in-flight HTTP requests
- MQ: `grpcServer.GracefulStop()` + `broker.Close()` — flushes all WALs
- Collector: `c.Close()` closes gRPC conn; pgx pool flushes
- Streamer: `pub.Close()` calls `stream.CloseSend()` then closes connection

### 8.3 Idempotency — three layers

1. **Streamer** is naturally idempotent — same CSV row gives same proto fields (modulo timestamp)
2. **MQ** is at-least-once — broker may re-deliver if consumer crashes pre-ack
3. **Collector** uses `ON CONFLICT DO NOTHING` — duplicates silently dropped at the DB

The `(uuid, metric_name, sample_at)` natural key is what makes layer 3 work without app-level coordination.

### 8.4 The interesting concurrency points

- **One mutex per partition** — never per topic. Partition 0 publishes don't block partition 1 publishes.
- **Coalesced subscriber notifications** — the `notify chan struct{}` has buffer 1; multiple publishes collapse into one wake-up.
- **One subscriber struct registered on N partitions** — a publish to ANY of a consumer's owned partitions wakes the same channel, so the gRPC handler has a single `select` case for "data available" instead of N.
- **`done` channels for rebalance signalling** — close-once semantics let the broker tell a member "reconnect" without the member having to poll group state.

### 8.5 What WAL gives you in practice

Scenario: Broker pod crashes after publishing offset 47 to partition 0 but before flushing the buffer.
- The OS may or may not have written to disk yet (depends on whether `syncBytes` was hit).
- If not flushed: offset 47 is lost. Streamer's `Publish` did get a response with `offset: 47`, so the streamer thinks it succeeded.
- If flushed: offset 47 is on disk; on restart, `replayWAL` reads it and reseeds ring buffer with `head = 48`.

For stronger durability, set `MQ_WAL_SYNC_BYTES=0` (sync every record). Pays in throughput. Default 4096 is a reasonable middle.

### 8.6 The observability sidecar

A small [`obs` package](../services/messagequeue/internal/obs/obs.go) (copied into MQ, Streamer, Collector) starts an HTTP server on `METRICS_PORT` (default 9091) that exposes:

| Path | Behaviour |
|---|---|
| `/healthz` | Always returns 200 — used by Kubernetes liveness probes. Answers immediately, even while the service is waiting for an upstream dependency. |
| `/readyz` | Calls a `ReadyFunc` supplied by main.go. Collector wires it to `db.Ping`; MQ + Streamer have no real readiness signal yet so they return 200 unconditionally. |
| `/metrics` | Default Prometheus collectors (Go runtime, process). Domain counters are not yet wired — documented as a follow-up. |

The Gateway has its own HTTP server (the REST API), so it serves these routes directly through the chi router rather than via a sidecar. `/swagger/*` lives on the same port.

## 9. Resilient startup — any order, any machine

Every service tolerates its dependencies being down at startup. Instead of crash-looping until Kubernetes gives up, each one **waits politely with exponential backoff** until the dependency is reachable.

### 9.1 The retry helper

Three copies live in the codebase, structurally identical:
- [services/streamer/internal/publisher/retry.go](../services/streamer/internal/publisher/retry.go)
- [services/collector/internal/store/retry.go](../services/collector/internal/store/retry.go)
- [services/gateway/internal/store/retry.go](../services/gateway/internal/store/retry.go)

```go
func retryWithBackoff(ctx context.Context, logger *zap.Logger, label string, fn func(context.Context) error) error {
    backoff := 200 * time.Millisecond
    const maxWait = 10 * time.Second
    for {
        if err := fn(ctx); err == nil { return nil }
        // ... log "waiting for dependency", select on ctx.Done() vs time.After(backoff),
        //     then double backoff up to maxWait
    }
}
```

Three packages → three copies → ~90 lines of duplication. Centralising would mean a shared utility module that's awkward in the workspace layout. Standard Go practice at this scale.

### 9.2 Where it's used

| Service | What it waits for | File |
|---|---|---|
| Streamer | MQ broker — wraps `CreateTopic` + `Publish` stream open | [publisher.go](../services/streamer/internal/publisher/publisher.go) |
| Collector | Postgres — wraps `pool.Ping` | [collector store/postgres.go](../services/collector/internal/store/postgres.go) |
| Collector | MQ broker — `runOnce` reconnect loop, already in [consumer.go:71-83](../services/collector/internal/consumer/consumer.go#L71) |
| Gateway | Postgres — wraps `pool.Ping` | [gateway store/postgres.go](../services/gateway/internal/store/postgres.go) |

### 9.3 What you see in logs during a wait

```json
{"level":"warn","msg":"waiting for dependency","op":"CreateTopic","attempt":1,"next_retry_in":"200ms","error":"connection refused"}
{"level":"warn","msg":"waiting for dependency","op":"CreateTopic","attempt":2,"next_retry_in":"400ms","error":"connection refused"}
...
{"level":"info","msg":"dependency reachable","op":"CreateTopic","attempts":7}
```

The operator immediately sees *which* dependency is missing, how long it's been waiting, and exactly when the wait ended.

### 9.4 The signal-cancellation contract

The retry loop selects on `ctx.Done()`, so SIGTERM during the wait exits cleanly rather than blocking shutdown. The streamer's main.go starts the `obs` sidecar FIRST so liveness probes pass during the dependency wait:

```go
// streamer/cmd/server/main.go
obsServer := obs.Start(cfg.MetricsPort, logger, nil)    // /healthz answers immediately
pub, err := publisher.New(ctx, cfg, logger)             // may block waiting for MQ
```

If a kubelet sends SIGTERM while still waiting for MQ, the ctx cancels, `publisher.New` returns `context.Canceled`, main treats it as a clean shutdown, and the pod terminates without ever spamming alarming "CrashLoopBackOff" events.

## 10. Multi-cluster topology

Every service can be installed in any subset of clusters. Three escape hatches make cross-cluster wiring work:

| Knob | Used when | What it does |
|---|---|---|
| `messagequeue.enabled = false` | This cluster does NOT run the broker | All MQ resources (StatefulSet, Service, PVC) are skipped |
| `messagequeue.externalAddress` | This cluster needs to TALK to a broker elsewhere | `mqAddress` helper returns this `host:port` instead of the in-cluster DNS name |
| `messagequeue.service.type = LoadBalancer` (or `NodePort`) | This cluster runs the broker AND needs to expose it | Renders a second `*-external` Service alongside the headless one |
| `postgresql.enabled = false` + `externalDatabase.*` | This cluster uses a Postgres elsewhere | DSN built from `externalDatabase` instead of the Bitnami subchart |

### 10.1 The `mqAddress` helper (the resolution rule)

[deploy/helm/gpu-telemetry/templates/_helpers.tpl](../deploy/helm/gpu-telemetry/templates/_helpers.tpl):

```go
{{- define "gpu-telemetry.mqAddress" -}}
{{- if .Values.messagequeue.externalAddress }}
{{- .Values.messagequeue.externalAddress }}            // 1. explicit external wins
{{- else if .Values.messagequeue.enabled }}
{{- printf "%s-messagequeue:%d" (...) (...) }}         // 2. in-cluster DNS
{{- else }}
{{- fail "..." }}                                       // 3. neither → fail fast
{{- end }}
{{- end }}
```

Both the streamer and the collector use this same helper to populate their `MQ_ADDRESS` env var. One knob → both services in sync, no chance of misconfiguration.

### 10.2 Four canonical topologies + example values

The chart ships four example values files in [deploy/helm/gpu-telemetry/examples/](../deploy/helm/gpu-telemetry/examples/) covering:

| File | Cluster role |
|---|---|
| `messagequeue-only.values.yaml` | Just the broker, LoadBalancer-exposed |
| `streamer-only.values.yaml` | Just the streamer, pointing at external MQ |
| `collector-only.values.yaml` | Collector + Postgres, external MQ |
| `gateway-only.values.yaml` | Gateway + Ingress, external Postgres |

Full installation procedure in **[docs/MULTI_CLUSTER.md](MULTI_CLUSTER.md)** — including the LoadBalancer endpoint discovery, the streamer.csvData flow, and per-cluster sanity checks.

### 10.3 Why streamer + collector don't have their own `externalAddress`

Direction of connection. Both streamer and collector are pure **clients** of the broker — they OPEN the gRPC connection, the broker pushes data over the already-open socket. Nothing ever dials into a streamer or collector pod (Prometheus scrape and kubelet probes are cluster-local).

Server-side components (MQ broker, Postgres, Gateway) get `externalAddress`/`externalDatabase.host`/`ingress.host` respectively. Client-side components (Streamer, Collector) don't, because they have nothing to be reached at.
